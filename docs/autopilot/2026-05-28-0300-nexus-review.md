# Nexus Review — Secret Manager Live Probe Re-review (2026-05-28 03:08 CEST)

## Verdict: GO with follow-up findings

The previous blocking Nexus findings are addressed for this uncommitted change. The live probe route now fails closed when the service audit client is unavailable before any provider probe, applies an endpoint-specific `secretManagerLiveProbe` rate-limit policy scoped by authenticated user + connector, and checks `persistActionBridgeControlAuditEvent` before returning successful evidence. Raw secret refs, provider tokens, secret values, and provider resource names are not returned by the route; responses use digest-only/redacted summaries.

This is acceptable as a controlled ActionBridge ops/control-plane evidence route, with the production Secret Manager rollout still correctly blocked on real environment provisioning, least-privilege service identity/token issuance, live evidence, and Sentinel release review.

## Connector type
ActionBridge ops/control-plane route for webhook connector managed-secret live-access evidence.

## Supported actions
- `read` / verification only: probes the server-owned managed secret reference for an existing HMAC webhook connector.
- No connector mutation.
- No raw secret disclosure.

## Required auth/session
- Authenticated Supabase user session.
- Connector lookup is owner-scoped with `.eq('user_id', user!.id)` and `.eq('id', connectorId)`.
- Route only proceeds for webhook connectors in `hmac_sha256` mode with a stored `secret_ref`.
- Service audit client is required before probe execution.

## Risk per action
- Medium-high: each accepted call can cause a Google Secret Manager REST `versions/latest:access` attempt.
- Mitigations now present:
  - endpoint-specific rate limit: `secretManagerLiveProbe`;
  - discriminator includes user id + connector id;
  - production distributed rate-limit mode still fails closed without trusted proxy/distributed store;
  - audit persistence is required for evidence responses;
  - output is redacted/digest-only.

## Sentinel policy references
- `sentinel.actionbridge.secret_manager.live_probe.v1`
- Audit event names:
  - `secret_manager.live_probe_verified`
  - `secret_manager.live_probe_failed`
- Audit result includes `controlPlane: true`, `networkExecution: false`, digest-only connector/secret/rate-limit references, and `redacted: true`.

## Remaining actionable findings

### 1. Medium — route tests are still marker/regex-heavy
`test-actionbridge-behavioral-modules.mjs` verifies an important source pattern, but it does not execute the route contract.

Action:
- Add route-level tests with mocked auth Supabase, service Supabase, rate-limit/probe paths, and audit persistence for:
  - unauthenticated `401`;
  - missing connector id `400`;
  - owner-scoped not-found `404`;
  - non-webhook/non-HMAC rejection `409`;
  - service audit client unavailable `503` before probe;
  - rate-limited `429` before probe;
  - audit persistence failure `503` after probe;
  - success `200` and denied/probe failure `409`;
  - negative redaction assertions for raw `secret_ref`, token values, secret values, and provider resource names.

### 2. Medium — spec documentation for the live-probe route appears incomplete
The checklist/blocker docs mention the route, and `scripts/test-actionbridge-contracts.mjs` was expanded to require live-probe spec tokens. The spec itself should explicitly document the route contract so operators/Sentinel know what evidence means.

Action:
- Extend `docs/specs/actionbridge-webhook-secret-bootstrap-rotation.md` with a short “Managed Secret Live Probe Evidence” section covering:
  - `POST /api/actionbridge/ops/secret-manager-live-probe`;
  - request shape (`connectorId` / `connector_id`);
  - auth/owner scope;
  - rate-limit policy `secretManagerLiveProbe`;
  - Sentinel policy marker;
  - audit event names;
  - fail-closed semantics for audit unavailable/audit failed/rate limited;
  - redacted response shape.

### 3. Low — successful evidence can still represent provider access followed by failed audit
The route now returns `503 ACTIONBRIDGE_SECRET_MANAGER_PROBE_AUDIT_FAILED` when persistence fails, which prevents a false positive success. However, the provider access already happened before the failed audit write.

Action:
- If Sentinel requires a durable record for every provider access attempt, add a pre-probe `pending` audit event and then a final success/failure event, or use an append-only evidence transaction pattern. For current controlled evidence capture, the existing fail-closed response is acceptable.
