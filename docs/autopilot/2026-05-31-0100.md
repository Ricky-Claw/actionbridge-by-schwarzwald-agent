## Blocker selected
Bridge-handshake CORS/origin hardening still needed explicit negative route coverage and a fail-closed POST path for missing/invalid browser `Origin`. Without this, a leaked setup token could be tested from non-browser contexts without the same customer-origin proof that the bridge script normally provides.

## What changed
- `POST /api/actionbridge/bridge/handshake` now normalizes the request `Origin` through the same public-HTTPS origin guard used for CORS.
- Missing or invalid `Origin` fails closed with `INVALID_ACTIONBRIDGE_BRIDGE_ORIGIN` and no `Access-Control-Allow-Origin`.
- Body origin must match the normalized request `Origin`; mismatch fails before DB lookup.
- Existing CORS behavior stays bounded: valid public HTTPS origins are echoed for preflight, normal denials, and rate-limit denials; no credentials are allowed.
- Userflow smoke now checks:
  - valid preflight CORS;
  - invalid/private/http Origin preflight denial without CORS;
  - missing-Origin POST denial without CORS;
  - Origin/body mismatch denial with requester-origin CORS;
  - invalid-token denial with CORS;
  - bridge-handshake rate-limit denial with CORS and `Retry-After`.
- Contract marker updated to require the new normalized Origin-header gate.
