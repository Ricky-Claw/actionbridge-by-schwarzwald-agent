## Blocker selected
Setup-session DNS/meta/`.well-known` domain verification had moved to real customer self-setup, but its HTTP verification path still used `fetch()` after DNS validation. That reopened DNS rebinding/TOCTOU risk and lacked the same pinned-connection discipline already used by Webhook-v1 delivery.

## What changed
- Replaced setup-domain HTTP verification `fetch()` with connection-pinned HTTPS GET:
  - resolve once with an app-level DNS timeout;
  - validate every returned DNS answer through `decideActionBridgeDnsPinning` before any HTTP request;
  - connect to the selected validated IP while preserving original Host and SNI;
  - disable agent socket reuse for this probe;
  - block redirects by destroying the response/request without body drain;
  - enforce a hard wall-clock HTTP deadline plus socket timeout;
  - cap streamed response bytes before matcher parsing.
- Added bounded DNS TXT lookup handling with structured fail-closed evidence.
- Added explicit redacted evidence markers for DNS-vs-HTTP probe semantics (`dnsLookupAttempted`, `httpRequestAttempted`, timeout reasons).
- Extended contract/security gates so domain verification must keep pinned HTTPS markers, DNS/TXT/HTTP timeouts, no `fetch(`, redirect destruction, byte cap, and audit evidence markers.
