# Nexus Review — ActionBridge Webhook Secret Resolver — 2026-05-26 03:00 Europe/Berlin

## Decision
**NO-GO** for marking the production secret-manager/KMS blocker closed.

The execute-path implementation is directionally safe and the full local gate is green, but the current uncommitted slice has production-rollout and docs-accuracy blockers around managed-secret rotation/operator flow and secret-ref mapping assumptions.

## Blocking implementation / architecture issues
1. **Managed-secret mode breaks the existing operator rotation route.**
   - Evidence: `src/frontend/app/api/actionbridge/ops/webhook-secret-rotation/route.ts:148` still calls the synchronous `resolveActionBridgeWebhookSigningSecret(...)`.
   - Evidence: `src/frontend/lib/actionbridge/webhook-signing.ts:155-156` makes the synchronous resolver fail closed with `secret_manager_async_required` whenever `ACTIONBRIDGE_SECRET_MANAGER_PROVIDER=google_secret_manager_rest`.
   - Result: in the documented production managed mode, the rotation route cannot validate/apply a next managed secret ref. That contradicts the docs’ claim that the rotation route “requires server-side secret-ref resolution” and is production-rollout-adjacent.

2. **Secret-ref → Google Secret Manager secret-id mapping is under-specified and likely incompatible for some valid ActionBridge refs.**
   - Evidence: `normalizeSecretRef()` accepts labels containing dots and colons after the prefix via `[a-zA-Z0-9._:-]{8,160}` (`webhook-signing.ts:20-22`).
   - Evidence: `secretIdFromRef()` preserves dots and only replaces characters outside `[a-zA-Z0-9._-]` (`webhook-signing.ts:43-44`).
   - Risk: Google Secret Manager secret IDs are stricter than this in common deployments; refs that pass ActionBridge validation may map to inaccessible/invalid provider IDs. Either narrow the ref grammar for managed mode, fully document the exact provider-safe subset, or encode/map refs through a deterministic provider-safe ID scheme.

3. **Docs close the production KMS/secret-manager blocker too strongly.**
   - Evidence: `docs/production-readiness-checklist.md:25` changes “Real secret manager/KMS integration for production” to checked.
   - Evidence: `docs/sentinel-production-blockers.md` marks the production KMS/secret-manager resolver with access audit checked.
   - Concern: the code adds a direct REST resolver with a bearer token from env (`webhook-signing.ts:51-64`), but remaining hardening still includes scoped service identity/least-privilege token issuance (`docs/specs/actionbridge-webhook-secret-bootstrap-rotation.md:98-100`) and the rotation route is not managed-mode compatible. Keep this as partial/provisional until those gaps are resolved or explicitly scoped out.

## Secret-ref mapping assumptions
- Current mapping: `actionbridge:webhook-signing:<label>` → `<label>` with unsafe chars replaced by `-`, max 160 chars (`webhook-signing.ts:43-44`).
- Safe assumption not yet proven: every syntactically valid ActionBridge secret ref maps to a valid Google Secret Manager secret id. This needs either tests or a narrower managed-mode validator.
- Positive: summaries use only `secretRefDigest` and optional `versionResourceDigest`; no raw ref/secret/token is intentionally returned (`webhook-signing.ts:54-79`, `83-93`).

## Timeout / fetch behavior
- Positive: managed resolver uses `cache: 'no-store'` and `AbortSignal.timeout(3000)` (`webhook-signing.ts:60-64`).
- Positive: non-2xx responses fail closed with redacted audit metadata (`webhook-signing.ts:66-67`), and provider exceptions are caught by the async resolver (`webhook-signing.ts:119-128`).
- Watch item: no explicit response body size/content-type guard before `response.json()`. Google Secret Manager is trusted-ish, but this should be noted if the resolver pattern is generalized.
