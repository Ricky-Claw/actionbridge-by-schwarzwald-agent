## Blocker selected
Customer setup-session bridge snippets were still hardcoded to `https://actionbridge.schwarzwald-agent.de` for `bridge.js` and the handshake endpoint. That can break deployed staging smoke tests and can route setup-token bridge traffic to the wrong ActionBridge environment instead of the customer-approved deployment.

## Attack hypothesis
If staging or another approved ActionBridge deployment emits a customer setup snippet pinned to production, the customer installs a script that posts the setup-token handshake to the wrong environment. Impact is mainly cross-environment confusion/token-boundary drift, not broad account takeover, because setup tokens remain placeholder/masked in the UI and handshake still enforces origin binding.

## What changed
- Added server-side public bridge-origin resolution in `src/frontend/lib/actionbridge/setup-session.ts`:
  - `ACTIONBRIDGE_PUBLIC_BASE_URL` is accepted only as an exact HTTPS origin.
  - Userinfo, path, query/hash, unsafe/private hosts, trimmed/lenient URLs, and public request-derived origins are rejected.
  - Request-derived localhost HTTP is allowed only for non-production development.
  - Missing/invalid config falls back to the canonical production ActionBridge origin.
- `/api/actionbridge/setup-session` now resolves and injects the sanitized `bridgePublicOrigin` into the setup-session view.
- Customer setup UI now displays the Bridge-Origin next to the masked snippet.
- Contract/security/behavioral/demo tests now cover configured-origin snippets and the dev/public-origin rejection path.
- Smoke/readiness/gate docs now require `ACTIONBRIDGE_PUBLIC_BASE_URL` to be pinned for deployed staging/production smoke runs.

## Severity
Medium — cross-environment setup-token boundary/configuration issue. Not Critical/High because it does not bypass origin binding, does not expose token digests/secrets, and bridge execution remains connected-only/catalog-safe.

## Safe reproduction summary
1. Create or load a setup session in a non-production deployment.
2. Without this fix, inspect the customer snippet: `src` and `data-endpoint` point at the production ActionBridge origin.
3. With this fix, set `ACTIONBRIDGE_PUBLIC_BASE_URL=https://staging.actionbridge.example`; the emitted snippet pins both `src` and `data-endpoint` to that staging origin.
4. Try unsafe values such as `https://localhost`, `https://token@example.test`, `https://example.test/path`, `https:example.test`, or dev request origin `https://evil.example`; they are rejected/fall back safely.

## What would be possible if exploited
A staging/customer setup could silently install a bridge script that posts the setup-token handshake to the wrong ActionBridge environment, causing failed onboarding, polluted production/staging state, or confusing audit evidence. With a permissive non-prod host setup, request-derived public origins could also have caused snippets to point at an attacker-controlled public origin; this run closed that path.

## Sentinel control required
- Deployment/config guardrail: `ACTIONBRIDGE_PUBLIC_BASE_URL` must be set to the exact approved ActionBridge HTTPS origin for staging/production.
- Release smoke must assert snippet `src` and `data-endpoint` match the approved environment.
- Future hardening: add preflight/fail-closed config check for deployed non-prod environments instead of relying on fallback.
