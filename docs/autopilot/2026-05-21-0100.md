## Blocker selected
High/Critical operator alerts were durable and owner-scoped, but the operator API only listed alerts. There was no dedicated alert acknowledgement/resolution mutation path with audit evidence, leaving operational lifecycle handling weaker than the existing error-log lifecycle.

## Changes
- Added `PATCH /api/actionbridge/alerts` for authenticated owner-scoped acknowledgement/resolution of High/Critical operator alerts.
- Enforced monotonic status transitions (`open` → `acknowledged` → `resolved`) and compare-and-set status predicates to block stale downgrade races.
- Used service mutation only after owner-scoped user lookup.
- Added `update_actionbridge_operator_alert_status` RPC so alert status and backing error-log status transition atomically in one database transaction.
- Preserved `acknowledged_at`/`resolved_at` semantics and avoided reopening resolved rows.
- Added `operator_alert.status_changed` control audit events with redacted, bounded metadata.
- Extended behavioral module coverage for route guard tokens, alert lifecycle race proof, and RPC migration guard tokens.
