## Blocker selected
Bound customer setup links exposed verification-method choices in the setup UI, but the public setup session could not issue/check real DNS TXT, meta tag, or `.well-known` domain verification challenges. That blocked customer self-setup for origin-locked website bridge flows.

## What changed
- Added token-scoped setup verification route: `POST/PATCH /api/actionbridge/setup-session/verification`.
- Bound setup links can issue/check real domain challenges without browser auth.
- Unbound setup links stay bridge-preview only via `canIssueVerificationChallenge`.
- Rechecks setup-token digest, link usability, connector origin binding, connector state, and durable quarantine before challenge/activation.
- Allows activation only from draft connectors; active/pass is idempotent no-op.
- Denies disabled, paused, revoked, failed, quarantined, mismatched, expired, unbound, or non-owner connector states.
- Prevents verified-row churn: already-verified rows do not get overwritten and can retry connector activation/no-op safely.
- Keeps `network_execution_enabled=false`; no setup-token route can grant live execution.
- Control audit can now record real verification network probes with `networkExecution:true`.
